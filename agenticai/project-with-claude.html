June 2026  ·  Peerapat A  ·  ← Agentic AI

Most people start using an AI agent the same way: they open a chat, paste some code, and explain everything from scratch — the stack, the conventions, the way the team likes to work. Then they do it again the next day. And the next. The agent is capable, but it has no memory of your project and no idea how your team actually works.

Running a project with Claude well comes down to two artifacts that solve exactly those two gaps:

  • AGENTS.md — the context: who we are, what we are building, and the rules of the house.
  • Skills — the process: the repeatable steps the team follows to get work done.

Think of it as hiring a very fast engineer. AGENTS.md is the onboarding doc you hand them on day one. Skills are the runbooks they reach for when doing a specific job. This post walks through both, using a normal software development workflow as the example.

AGENTS.md — the Project Constitution

AGENTS.md is a plain Markdown file at the root of your repository. Agents read it automatically and treat it as standing context for every task in that project. It is not a place for documentation that lives elsewhere — it is the short list of things the agent would otherwise get wrong: your stack, your commands, your conventions, and your hard rules.

Keep it tight. A good AGENTS.md for a software project answers four questions:

  • What is this? — one paragraph on the product and the architecture.
  • How do I run it? — the exact build, test, and lint commands.
  • What are the conventions? — language, style, naming, folder layout.
  • What must never happen? — the non-negotiable rules.

Here is a realistic example for a backend service:

# AGENTS.md

## Project
Order service for an e-commerce platform. Scala + Play Framework,
PostgreSQL via Slick, deployed to AWS ECS. REST API consumed by a
React front end.

## Commands
- Build:   sbt compile
- Test:    sbt test
- Lint:    sbt scalafmtCheckAll
- Run:     sbt run   (port 9000)

## Conventions
- Controllers stay thin; business logic lives in services.
- One file per public class. Package by feature, not by layer.
- Every new endpoint ships with a PlaySpec integration test.
- Use Option, never null. No throwing for control flow.

## Rules
- Never commit secrets; config comes from environment variables.
- Do not edit files under db/migrations that are already applied.
- Open a feature branch; never push straight to main.
- If a test fails, stop and report — do not delete or skip it.

Notice what is not here: no tutorial on Scala, no copy of the README. The agent already knows the language. AGENTS.md only carries the things specific to this project that change its behaviour. The shorter and sharper it is, the more reliably it gets followed.

Skills — the Work Process

AGENTS.md tells the agent where it is. A Skill tells it how to do a specific job. A skill is a Markdown file with a small frontmatter header and a body of step-by-step instructions. When you invoke it, the agent loads that process and follows it like a checklist — so a task runs the same way whether it is Monday morning or Friday at 5pm, whether it is you driving or a teammate.

The trick to managing a project is to stop thinking in one-off prompts and start thinking in processes. Every recurring activity in your team — building a feature, reviewing a PR, cutting a release — is a process you already follow informally. Write it down once as a skill, and the agent executes it consistently forever.

Example: A Feature-Development Skill

Take the most common workflow on any dev team: turning a ticket into merged code. Most teams have an unwritten version of this in their heads. Here it is made explicit: 

---
name: feature-dev
description: "Implements a feature ticket end to end: branch, code,
  test, and prepare a pull request following team conventions."
argument-hint: "Provide the ticket ID and a short description of the
  feature to build."
---

# Feature Development Workflow

You are a senior engineer on this project. Follow the team's process
exactly. Do not skip steps and do not push to main.

## Steps
1. Restate the ticket in your own words and list acceptance criteria.
   Ask for clarification if any criterion is ambiguous.
2. Create a branch named `feature/<ticket-id>-<slug>`.
3. Locate the relevant service and controller before writing code.
   Reuse existing patterns; do not introduce new libraries.
4. Implement the change. Keep controllers thin — logic goes in services.
5. Write a PlaySpec integration test covering every acceptance criterion,
   including at least one failure case.
6. Run `sbt scalafmtCheckAll` and `sbt test`. If anything fails, fix it
   before continuing.
7. Summarise the change: what, why, files touched, and how it was tested.
   Prepare a PR description from that summary.

## Definition of Done
- All acceptance criteria are covered by a passing test.
- Lint and the full test suite are green.
- No secrets, no edits to applied migrations, no direct push to main.

This single file encodes branch naming, the thin-controller rule, the testing standard, and a definition of done. Invoke it with a ticket and the agent produces work that already matches how your team operates — instead of generic code you then have to reshape.

Composing Skills into a Pipeline

One skill is useful; a small library of them turns the agent into a teammate that knows your whole delivery process. A typical software project ends up with a handful:

  • feature-dev — ticket → branch → code → tests → PR (above).
  • code-review — read a diff and flag bugs, missing tests, and convention breaks.
  • bugfix — reproduce with a failing test first, then fix until it passes.
  • release — bump version, generate the changelog, tag, and draft release notes.

Each skill is a stage in your pipeline. Because every stage reads the same AGENTS.md, they all share one definition of "correct" — the conventions and rules live in exactly one place. Update a rule in AGENTS.md and every skill inherits it on the next run.

How It Works in Practice

A day of project work then looks like this:

$ claude

> /feature-dev PROJ-218 "add idempotency key to POST /orders"

  ✓ Restated ticket, listed 4 acceptance criteria
  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore
  ✓ Wrote OrdersControllerSpec (4 cases + duplicate-key case)
  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

> /code-review

  ⚠ Missing test: concurrent requests with the same key
  ✓ Conventions OK · no secrets · migration untouched

You stayed at the level of intent — "add an idempotency key" — and the agent handled the mechanics the way your team would. The review skill even caught a gap the feature skill missed, because the two processes encode different responsibilities.

Why This Beats Prompting

  • ✅ Consistency — the process runs the same way for everyone, every time.
  • ✅ Onboarding — a new hire (human or agent) is productive by reading two files.
  • ✅ Single source of truth — rules live in AGENTS.md, not in scattered chat history.
  • ✅ Living documentation — your AGENTS.md and skills are how the team works, kept honest because they are executed daily.
  • ✅ Scales with the team — commit them to the repo and the whole process is version-controlled.

Bottom Line

You do not manage a project with an AI agent by prompting harder. You manage it by writing down two things: who we are (AGENTS.md) and how we work (Skills). Once those exist in the repo, every task starts with the agent already knowing your stack, your standards, and your process — and you get to spend your attention on intent instead of instructions.

Start small: write the AGENTS.md today, then turn your single most repeated workflow into a skill. Add the next one when you notice yourself explaining the same process twice.
