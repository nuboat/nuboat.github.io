ใช้งาน Claude ผ่าน AGENTS และ SKILL

June 2026  ·  Peerapat A  ·  ← Agentic AI

EN TH ZH JA DE

Most people start using an AI agent the same way: they open a chat, paste some code, and explain everything from scratch — the stack, the conventions, the way the team likes to work. Then they do it again the next day. And the next. The agent is capable, but it has no memory of your project and no idea how your team actually works.

Running a project with Claude well comes down to two artifacts that solve exactly those two gaps:

  • AGENTS.md — the context: who we are, what we are building, and the rules of the house.
  • Skills — the process: the repeatable steps the team follows to get work done.

Think of it as hiring a very fast engineer. AGENTS.md is the onboarding doc you hand them on day one. Skills are the runbooks they reach for when doing a specific job. This post walks through both, using a normal software development workflow as the example.

AGENTS.md — the Project Constitution

AGENTS.md is a plain Markdown file at the root of your repository. Agents read it automatically and treat it as standing context for every task in that project. It is not a place for documentation that lives elsewhere — it is the short list of things the agent would otherwise get wrong: your stack, your commands, your conventions, and your hard rules.

Keep it tight. A good AGENTS.md for a software project answers four questions:

  • What is this? — one paragraph on the product and the architecture.
  • How do I run it? — the exact build, test, and lint commands.
  • What are the conventions? — language, style, naming, folder layout.
  • What must never happen? — the non-negotiable rules.

Here is a realistic example for a backend service:

# AGENTS.md

## Project
Order service for an e-commerce platform. Scala + Play Framework,
PostgreSQL via Slick, deployed to AWS ECS. REST API consumed by a
React front end.

## Commands
- Build:   sbt compile
- Test:    sbt test
- Lint:    sbt scalafmtCheckAll
- Run:     sbt run   (port 9000)

## Conventions
- Controllers stay thin; business logic lives in services.
- One file per public class. Package by feature, not by layer.
- Every new endpoint ships with a PlaySpec integration test.
- Use Option, never null. No throwing for control flow.

## Rules
- Never commit secrets; config comes from environment variables.
- Do not edit files under db/migrations that are already applied.
- Open a feature branch; never push straight to main.
- If a test fails, stop and report — do not delete or skip it.

Notice what is not here: no tutorial on Scala, no copy of the README. The agent already knows the language. AGENTS.md only carries the things specific to this project that change its behaviour. The shorter and sharper it is, the more reliably it gets followed.

Skills — the Work Process

AGENTS.md tells the agent where it is. A Skill tells it how to do a specific job. A skill is a Markdown file with a small frontmatter header and a body of step-by-step instructions. When you invoke it, the agent loads that process and follows it like a checklist — so a task runs the same way whether it is Monday morning or Friday at 5pm, whether it is you driving or a teammate.

The trick to managing a project is to stop thinking in one-off prompts and start thinking in processes. Every recurring activity in your team — building a feature, reviewing a PR, cutting a release — is a process you already follow informally. Write it down once as a skill, and the agent executes it consistently forever.

Example: A Feature-Development Skill

Take the most common workflow on any dev team: turning a ticket into merged code. Most teams have an unwritten version of this in their heads. Here it is made explicit: 

---
name: feature-dev
description: "Implements a feature ticket end to end: branch, code,
  test, and prepare a pull request following team conventions."
argument-hint: "Provide the ticket ID and a short description of the
  feature to build."
---

# Feature Development Workflow

You are a senior engineer on this project. Follow the team's process
exactly. Do not skip steps and do not push to main.

## Steps
1. Restate the ticket in your own words and list acceptance criteria.
   Ask for clarification if any criterion is ambiguous.
2. Create a branch named `feature/<ticket-id>-<slug>`.
3. Locate the relevant service and controller before writing code.
   Reuse existing patterns; do not introduce new libraries.
4. Implement the change. Keep controllers thin — logic goes in services.
5. Write a PlaySpec integration test covering every acceptance criterion,
   including at least one failure case.
6. Run `sbt scalafmtCheckAll` and `sbt test`. If anything fails, fix it
   before continuing.
7. Summarise the change: what, why, files touched, and how it was tested.
   Prepare a PR description from that summary.

## Definition of Done
- All acceptance criteria are covered by a passing test.
- Lint and the full test suite are green.
- No secrets, no edits to applied migrations, no direct push to main.

This single file encodes branch naming, the thin-controller rule, the testing standard, and a definition of done. Invoke it with a ticket and the agent produces work that already matches how your team operates — instead of generic code you then have to reshape.

Composing Skills into a Pipeline

One skill is useful; a small library of them turns the agent into a teammate that knows your whole delivery process. A typical software project ends up with a handful:

  • feature-dev — ticket → branch → code → tests → PR (above).
  • code-review — read a diff and flag bugs, missing tests, and convention breaks.
  • bugfix — reproduce with a failing test first, then fix until it passes.
  • release — bump version, generate the changelog, tag, and draft release notes.

Each skill is a stage in your pipeline. Because every stage reads the same AGENTS.md, they all share one definition of "correct" — the conventions and rules live in exactly one place. Update a rule in AGENTS.md and every skill inherits it on the next run.

How It Works in Practice

A day of project work then looks like this:

$ claude

> /feature-dev PROJ-218 "add idempotency key to POST /orders"

  ✓ Restated ticket, listed 4 acceptance criteria
  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore
  ✓ Wrote OrdersControllerSpec (4 cases + duplicate-key case)
  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

> /code-review

  ⚠ Missing test: concurrent requests with the same key
  ✓ Conventions OK · no secrets · migration untouched

You stayed at the level of intent — "add an idempotency key" — and the agent handled the mechanics the way your team would. The review skill even caught a gap the feature skill missed, because the two processes encode different responsibilities.

Why This Beats Prompting

  • ✅ Consistency — the process runs the same way for everyone, every time.
  • ✅ Onboarding — a new hire (human or agent) is productive by reading two files.
  • ✅ Single source of truth — rules live in AGENTS.md, not in scattered chat history.
  • ✅ Living documentation — your AGENTS.md and skills are how the team works, kept honest because they are executed daily.
  • ✅ Scales with the team — commit them to the repo and the whole process is version-controlled.

Bottom Line

You do not manage a project with an AI agent by prompting harder. You manage it by writing down two things: who we are (AGENTS.md) and how we work (Skills). Once those exist in the repo, every task starts with the agent already knowing your stack, your standards, and your process — and you get to spend your attention on intent instead of instructions.

Start small: write the AGENTS.md today, then turn your single most repeated workflow into a skill. Add the next one when you notice yourself explaining the same process twice.

Skills — the Work Process

AGENTS.md จะบอก agent ว่า ตัวเองอยู่ที่ไหน แต่ Skill จะบอก วิธีทำงานที่เฉพาะเจาะจง Skill คือไฟล์ Markdown ที่มี header frontmatter เล็ก ๆ และมีคำแนะนำแบบ step-by-step อยู่ในส่วนเนื้อหา เมื่อคุณสั่ง invoke มัน agent จะโหลด process นั้นและทำตามเหมือนเป็น checklist — ดังนั้น ไม่ว่าจะเป็นเช้าวันจันทร์ หรือห้าโมงเย็นวันศุกร์ ไม่ว่าคุณจะเป็นคนสั่งงานเองหรือเป็นเพื่อนร่วมทีม งานก็จะถูกดำเนินการด้วยวิธีการเดียวกันเสมอ

เคล็ดลับในการจัดการโปรเจกต์คือการเลิกคิดในรูปแบบของการเขียน prompts ครั้งเดียวจบ แล้วเริ่มคิดเป็น processes กิจกรรมทุกอย่างที่เกิดขึ้นซ้ำ ๆ ในทีมของคุณ — ไม่ว่าจะเป็นการสร้าง feature, การ review PR, การตัด release — ล้วนเป็นกระบวนการที่คุณทำตามอย่างไม่เป็นทางการอยู่แล้ว แค่เขียนมันลงไปครั้งเดียวเป็น skill แล้ว agent จะ execute มันอย่างสม่ำเสมอตลอดไป

Example: A Feature-Development Skill

ยกตัวอย่าง workflow ที่พบบ่อยที่สุดใน dev team: การแปลง ticket ให้กลายเป็น merged code ทีมส่วนใหญ่มักจะมีเวอร์ชันที่ไม่ได้เขียนเป็นลายลักษณ์อักษรอยู่ในหัว นี่คือการทำให้มันชัดเจนขึ้น:

  test, and prepare a pull request following team conventions."
argument-hint: "Provide the ticket ID and a short description of the
  feature to build."
---

You are a senior engineer on this project. Follow the team's process
exactly. Do not skip steps and do not push to main.

## Steps
1. Restate the ticket in your own words and list acceptance criteria.

4. Implement the change. Keep controllers thin — logic goes in services.
5. Write a PlaySpec integration test covering every acceptance criterion,
   including at least one failure case.
6. Run `sbt scalafmtCheckAll` and `sbt test`. If anything fails, fix it
   before continuing.
7. Summarise the change: what, why, files touched, and how it was tested.

- All acceptance criteria are covered by a passing test.
- Lint and the full test suite are green.

ไฟล์เดียวนี้สามารถกำหนดทั้งวิธีการตั้งชื่อ branch, กฎ thin-controller, มาตรฐานการทดสอบ และ definition of done แค่สั่ง invoke ด้วย ticket แล้ว agent ก็จะสร้างผลงานที่สอดคล้องกับวิธีการทำงานของทีมคุณอยู่แล้ว — แทนที่จะได้โค้ดทั่วไปที่คุณต้องมาปรับแต่งเองทีหลัง

Composing Skills into a Pipeline

Skill เพียงหนึ่งอย่างก็มีประโยชน์แล้ว; แต่เมื่อคุณมี library รวม skill เหล่านี้ มันจะเปลี่ยน agent ให้กลายเป็นเพื่อนร่วมทีมที่รู้ process การ delivery ทั้งหมดของคุณ โดยทั่วไปแล้ว software project หนึ่งจะลงเอยด้วยการมีหลาย ๆ skills ด้วยกัน:

  • feature-dev — ticket → branch → code → tests → PR (ตามด้านบน)
  • code-review — อ่าน diff และจับจุดที่เป็น bugs, missing tests, และการขัดต่อ conventions ต่าง ๆ
  • bugfix — ทำซ้ำด้วยการเขียน failing test ก่อน จากนั้นแก้ไขจนกว่าจะ passes
  • release — อัปเดต version, สร้าง changelog, ทำ tag, และร่าง release notes

แต่ละ skill เปรียบเสมือนด่านหนึ่งใน pipeline ของคุณ เนื่องจากทุกด่านต่างอ่าน AGENTS.md ตัวเดียวกัน พวกมันทั้งหมดจึงใช้คำนิยามคำว่า "ถูกต้อง" ร่วมกัน — conventions และกฎต่าง ๆ จึงอยู่ในที่เดียวกันเท่านั้น หากมีการแก้ไขกฎใน AGENTS.md ทุก ๆ skill ก็จะสืบทอดกฎเหล่านั้นไปใช้ในการรันครั้งต่อไปทันที

How It Works in Practice

การทำงานของโปรเจกต์ในหนึ่งวันก็จะมีลักษณะประมาณนี้:

  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore

  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

คุณจะยังคงรักษาตัวเองให้อยู่ในระดับของ เจตนา (intent) — เช่น "เพิ่ม idempotency key" — แล้ว agent จะจัดการกับกลไกต่าง ๆ ด้วยวิธีที่ทีมของคุณน่าจะทำ นอกจากนี้ code-review skill ยังสามารถจับจุดที่ feature skill ทำพลาดไปได้ เพราะ process ทั้งสองถูกสร้างขึ้นมาเพื่อรับผิดชอบหน้าที่ที่ต่างกัน

Why This Beats Prompting

  • ✅ Consistency — process รันด้วยวิธีเดียวกันสำหรับทุกคนและทุกครั้ง
  • ✅ Onboarding — พนักงานใหม่ (ไม่ว่าจะเป็นมนุษย์หรือ agent) สามารถเริ่มทำงานได้เลยเพียงแค่อ่านไฟล์สองไฟล์นี้
  • ✅ Single source of truth — กฎต่าง ๆ จะอยู่ใน AGENTS.md ไม่ใช่กระจัดกระจายอยู่ในประวัติแชท
  • ✅ Living documentation — AGENTS.md และ skills ของคุณ คือ วิธีการทำงานจริง ๆ ของทีมคุณ ซึ่งจะถูกรักษาความถูกต้องอยู่เสมอเพราะพวกมันถูกนำมา execute ทุกวัน
  • ✅ Scales with the team — ทำการ commit พวกมันลง repo แล้ว process ทั้งหมดก็จะถูก version-controlled

Bottom Line

คุณไม่ได้บริหารโปรเจกต์ด้วย AI agent ด้วยการป้อน prompts อย่างหนักหน่วง คุณบริหารมันด้วยการเขียนสิ่งสองอย่างลงไป: เราคือใคร (AGENTS.md) และ เราทำงานกันอย่างไร (Skills) เมื่อสองสิ่งนี้มีอยู่ใน repo ทุก ๆ งานก็จะเริ่มต้นโดยที่ agent รู้จัก stack ของคุณ, มาตรฐานของคุณ, และ process ของคุณอยู่แล้ว — และคุณก็จะมีเวลาจดจ่ออยู่กับตัวเจตนา มากกว่าการต้องมานั่งสั่ง instructions ต่าง ๆ

เริ่มจากจุดเล็ก ๆ ก่อน: เขียน AGENTS.md ในวันนี้ จากนั้นเปลี่ยน workflow ที่คุณทำซ้ำบ่อยที่สุดให้กลายเป็น skill เพิ่ม skill ถัดไปเมื่อคุณสังเกตเห็นว่าตัวเองกำลังอธิบาย process เดิมซ้ำเป็นครั้งที่สอง

大多数人开始使用 AI agent 的方式是一样的：他们打开一个聊天窗口，粘贴一些代码，然后从零开始解释一切——stack、conventions、团队喜欢的工作方式。然后在第二天，他们又重复同样的过程。接着是第三天。AI agent 确实很有能力，但它没有关于 你的项目 (your project) 的记忆，也不知道 你的团队 (your team) 实际上是如何运作的。

要用 Claude 良好地运行一个项目，归结于可以准确解决这两个差距的两个 artifacts：

  • AGENTS.md — 上下文（context）：我们是谁、我们正在构建什么，以及这里的规则。
  • Skills — 流程（process）：团队为了完成工作所遵循的可重复步骤。

这就像雇佣一个速度极快的 engineer。AGENTS.md 是你在第一天递给他们的 onboarding 文档。而 Skills 是他们在执行特定任务时查阅的 runbooks。本文将以常规的 software development workflow 为例，带领你了解这两者。

AGENTS.md — 项目的宪法 (Project Constitution)

AGENTS.md 是放置在 repository 根目录下的一个纯 Markdown 文件。AI agents 会自动读取它，并将其作为该项目中所有任务的常驻上下文。它并不是用来存放已经存在于其他地方的文档的——它是一份简短清单，列出如果没人告诉 agent，它就会犯错的事情：你的 stack、你的 commands、你的 conventions 以及你的硬性规则。

保持简洁。一个针对 software project 的优秀 AGENTS.md 应该回答四个问题：

  • What is this? — 用一段话描述 product 和 architecture。
  • How do I run it? — 确切的 build、test 和 lint 命令。
  • What are the conventions? — 语言、风格、命名规则、文件夹结构。
  • What must never happen? — 绝对不可妥协的规则。

以下是一个真实的 backend service 示例：

# AGENTS.md

## Project
Order service for an e-commerce platform. Scala + Play Framework,
PostgreSQL via Slick, deployed to AWS ECS. REST API consumed by a
React front end.

## Commands
- Build:   sbt compile
- Test:    sbt test
- Lint:    sbt scalafmtCheckAll
- Run:     sbt run   (port 9000)

## Conventions
- Controllers stay thin; business logic lives in services.
- One file per public class. Package by feature, not by layer.
- Every new endpoint ships with a PlaySpec integration test.
- Use Option, never null. No throwing for control flow.

## Rules
- Never commit secrets; config comes from environment variables.
- Do not edit files under db/migrations that are already applied.
- Open a feature branch; never push straight to main.
- If a test fails, stop and report — do not delete or skip it.

注意这里 没有 什么：没有 Scala 的 tutorial，没有 README 的副本。因为 agent 已经懂这门语言了。AGENTS.md 只携带特定于 这个 项目的、会改变其行为的信息。它越简短精炼，被准确遵循的可能性就越大。

Skills — 工作流程 (Work Process)

AGENTS.md 告诉 agent 它在哪里。而 Skill 告诉它 如何完成特定的工作。Skill 是一个带有小型 frontmatter header 的 Markdown 文件，正文部分包含 step-by-step 的说明。当你 invoke 它时，agent 会加载那个 process 并像对照 checklist 一样遵循它——因此，无论是在周一早上还是在周五下午5点，无论是你亲自操作还是你的队友操作，任务的运行方式都将始终如一。

管理一个项目的诀窍在于，停止用一次性的 prompts 去思考，而是开始以 processes 的方式去思考。你团队中每一项重复出现的活动——无论是开发一个 feature、进行一次 PR review，还是切出一个 release——都是你已经在非正式地遵循的流程。将其作为 skill 写下来一次，agent 就会永远稳定地 execute 它。

示例：Feature-Development Skill

以任何 dev team 中最常见的 workflow 为例：将一个 ticket 转化为 merged code。大多数团队脑海中都有一个未成文的版本。这里将其明确化：

---
name: feature-dev
description: "Implements a feature ticket end to end: branch, code,
  test, and prepare a pull request following team conventions."
argument-hint: "Provide the ticket ID and a short description of the
  feature to build."
---

# Feature Development Workflow

You are a senior engineer on this project. Follow the team's process
exactly. Do not skip steps and do not push to main.

## Steps
1. Restate the ticket in your own words and list acceptance criteria.
   Ask for clarification if any criterion is ambiguous.
2. Create a branch named `feature/<ticket-id>-<slug>`.
3. Locate the relevant service and controller before writing code.
   Reuse existing patterns; do not introduce new libraries.
4. Implement the change. Keep controllers thin — logic goes in services.
5. Write a PlaySpec integration test covering every acceptance criterion,
   including at least one failure case.
6. Run `sbt scalafmtCheckAll` and `sbt test`. If anything fails, fix it
   before continuing.
7. Summarise the change: what, why, files touched, and how it was tested.
   Prepare a PR description from that summary.

## Definition of Done
- All acceptance criteria are covered by a passing test.
- Lint and the full test suite are green.
- No secrets, no edits to applied migrations, no direct push to main.

仅这一个文件就明确了 branch 命名规则、thin-controller 原则、测试标准以及 definition of done。通过提供一个 ticket 来 invoke 它，agent 就能产出符合你团队操作规范的工作成果——而不是生成一段你还得事后改造的通用代码。

将 Skills 组装成 Pipeline

一个 skill 就很有用了；而拥有一小套 skill 库，就可以把 agent 变成一个了解你们整个 delivery process 的队友。一个典型的 software project 最终会包含以下几个关键 skills：

  • feature-dev — ticket → branch → code → tests → PR（如上所示）。
  • code-review — 阅读 diff 并标记 bugs、缺失的 tests 以及违背 conventions 的地方。
  • bugfix — 首先通过一个 failing test 进行 reproduce，然后进行修复直到 passes。
  • release — 提升 version，生成 changelog，打上 tag，并起草 release notes。

每一个 skill 都是你 pipeline 中的一个阶段。因为每一个阶段读取的都是同一个 AGENTS.md，所以它们都共享同一种关于“正确”的定义——所有的 conventions 和规则都只存在于这一个地方。在 AGENTS.md 中更新一条规则，每一个 skill 都会在下一次运行时自动继承它。

在实践中它是如何运作的

一天的项目工作看起来会是这样的：

$ claude

> /feature-dev PROJ-218 "add idempotency key to POST /orders"

  ✓ Restated ticket, listed 4 acceptance criteria
  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore
  ✓ Wrote OrdersControllerSpec (4 cases + duplicate-key case)
  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

> /code-review

  ⚠ Missing test: concurrent requests with the same key
  ✓ Conventions OK · no secrets · migration untouched

你始终停留在 意图（intent） 层面——“添加一个 idempotency key”——而 agent 会按照你团队的方式处理具体的执行细节。code-review skill 甚至捕捉到了 feature skill 遗漏的问题，因为这两个 processes 各自承载着不同的职责。

为什么这比 Prompting 更好？

  • ✅ Consistency — process 对每个人、每一次，都以相同的方式运行。
  • ✅ Onboarding — 一位新员工（无论是人类还是 agent）只需阅读这两个文件就能立刻投入生产。
  • ✅ Single source of truth — 规则存在于 AGENTS.md 中，而不是散落在杂乱的聊天记录里。
  • ✅ Living documentation — 你的 AGENTS.md 和 skills 本身就是 团队的工作方式，由于每天都在被 execute，它们能够始终保持真实有效。
  • ✅ Scales with the team — 将它们 commit 到 repo 中，整个流程就会实现 version-controlled。

总结 (Bottom Line)

你不能通过拼命写更长的 prompts 来管理一个带有 AI agent 的项目。你需要通过写下两件事来管理它：我们是谁（AGENTS.md）和 我们是如何工作的（Skills）。一旦这两样东西存在于 repo 中，每一个任务在开始时，agent 就已经了解你的 stack、你的标准和你的 process——而你就可以把注意力放在意图上，而不是没完没了地提供 instructions。

从微小的行动开始：今天就写下 AGENTS.md，然后把你重复次数最多的 workflow 变成一个 skill。当你发现自己又在向它解释第二遍同样的 process 时，就增加下一个 skill 吧。

ほとんどの人は、AI agent の使い方を同じように始めます：チャットを開き、コードを貼り付け、すべてをゼロから説明します。stack、conventions、そしてチームの好む仕事の進め方などを。次の日にもまた同じことをし、その次の日も繰り返します。AI agent は有能ですが、あなたのプロジェクト に関する記憶を持たず、あなたのチーム が実際にどのように機能しているのかを知らないからです。

Claude を使ってプロジェクトをうまく進めるためには、まさにこれら2つのギャップを解決する2つの artifacts が重要になります：

  • AGENTS.md — 文脈（context）：私たちは誰で、何を構築していて、ここのルールは何か。
  • Skills — プロセス（process）：仕事を終わらせるためにチームが従う、反復可能なステップ。

これは、非常に速く仕事ができる engineer を雇うようなものだと考えてください。AGENTS.md は、初日に手渡す onboarding ドキュメントです。そして Skills は、彼らが特定の仕事をする際に参照する runbooks です。この記事では、通常の software development の workflow を例として、この両方について解説します。

AGENTS.md — プロジェクトの憲法 (Project Constitution)

AGENTS.md は、repository のルートにあるプレーンな Markdown ファイルです。AI agents はこれを自動的に読み込み、そのプロジェクトにおけるすべてのタスクの常時コンテキストとして扱います。ここは、他の場所にすでに存在するドキュメントを置く場所ではありません。これは、agent に伝えておかなければ間違えてしまうであろう事柄の短いリストです：あなたの stack、commands、conventions、そして絶対的なルールです。

簡潔に保ちましょう。software project に適した AGENTS.md は、次の4つの質問に答えます：

  • What is this? — product と architecture に関する1つのパラグラフ。
  • How do I run it? — 正確な build、test、および lint のコマンド。
  • What are the conventions? — 言語、スタイル、命名規則、フォルダ構成。
  • What must never happen? — 決して妥協できないルール。

以下は、backend service の現実的な例です：

Skills — 仕事のプロセス (Work Process)

AGENTS.md が agent に 自分がどこにいるのか を教えるのに対し、Skill は 特定の仕事をどうやって行うのか を教えます。Skill とは、小さな frontmatter のヘッダーと、step-by-step の指示が書かれた本文を持つ Markdown ファイルです。あなたがそれを invoke すると、agent はその process を読み込み、checklist のように従って実行します。したがって、月曜の朝であっても金曜の午後5時であっても、あなたが操作してもチームメイトが操作しても、タスクは常に同じように実行されます。

プロジェクト管理のコツは、単発の prompts で考えるのをやめ、processes で考え始めることです。feature の開発、PR の review、release の作成など、チーム内で繰り返されるすべての活動は、あなたがすでに非公式に従っているプロセスです。それを一度 skill として書き出せば、agent はそれを永遠に一貫して execute してくれます。

Skills を Pipeline に組み込む

1つの skill でも便利ですが、それらの小さなライブラリがあれば、agent はあなたの delivery process 全体を知るチームメイトに変わります。典型的な software project では、次のような一連の skills を持つことになります：

  • feature-dev — ticket → branch → code → tests → PR（上記参照）。
  • code-review — diff を読んで bugs、不足している tests、conventions 違反をフラグ付けする。
  • bugfix — 最初に failing test で問題を reproduce し、その後、passes するまで修正する。
  • release — version を上げ、changelog を生成し、tag を作成し、release notes の下書きをする。

各 skill は、あなたの pipeline における一つのステージです。すべてのステージが同じ AGENTS.md を読み込むため、それらはすべて「正しさ」の同じ定義を共有します。conventions とルールは、正確に一箇所にのみ存在します。AGENTS.md のルールを更新すれば、すべての skill は次の実行時にそのルールを継承します。

実践での動き方

そうすると、プロジェクトでの1日の作業は次のようになります：

$ claude

> /feature-dev PROJ-218 "add idempotency key to POST /orders"

  ✓ Restated ticket, listed 4 acceptance criteria
  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore
  ✓ Wrote OrdersControllerSpec (4 cases + duplicate-key case)
  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

  ✓ Conventions OK · no secrets · migration untouched

あなたは 意図（intent） のレベル —— 「idempotency key を追加する」 —— にとどまり、agent があなたのチームのやり方で機械的な作業を処理しました。code-review skill は、feature skill が見落としていたギャップさえも発見しました。なぜなら、2つの processes は異なる責任をコード化しているからです。

なぜこれが Prompting に勝るのか？

  • ✅ Consistency — process は誰に対しても、毎回同じように実行されます。
  • ✅ Onboarding — 新入社員（人間であれ agent であれ）は、2つのファイルを読むだけで生産的になれます。
  • ✅ Single source of truth — ルールは分散したチャット履歴の中ではなく、AGENTS.md に存在します。
  • ✅ Living documentation — あなたの AGENTS.md と skills それ自体 がチームの働き方であり、毎日 execute されることでその正確さが保たれます。
  • ✅ Scales with the team — それらを repo に commit すれば、プロセス全体が version-controlled されます。

まとめ (Bottom Line)

より懸命に prompts を書くことで、AI agent のいるプロジェクトを管理しようとしてはいけません。プロジェクトは、次の2つのことを書き留めることで管理します：私たちが誰であるか（AGENTS.md）と、私たちがどのように働くか（Skills）です。これらが repo に存在すれば、すべてのタスクは、agent がすでにあなたの stack、あなたの基準、そしてあなたの process を知っている状態からスタートします。そしてあなたは、instructions の代わりに意図（intent）に注意を注ぐことができるのです。

小さく始めましょう：今日 AGENTS.md を書き、最も繰り返されている workflow を一つの skill に変えてみてください。同じ process を二度説明していることに気づいた時が、次の skill を追加するタイミングです。

Die meisten Menschen beginnen auf die gleiche Weise mit einem AI agent zu arbeiten: Sie öffnen einen Chat, fügen etwas Code ein und erklären alles von Grund auf — den stack, die conventions, und die Art und Weise, wie das Team gerne arbeitet. Am nächsten Tag tun sie es wieder. Und am darauffolgenden Tag. Der AI agent ist zwar fähig, aber er hat keine Erinnerung an Ihr Projekt (your project) und keine Ahnung, wie Ihr Team (your team) eigentlich funktioniert.

Ein Projekt erfolgreich mit Claude zu führen, lässt sich auf zwei artifacts reduzieren, die genau diese beiden Lücken schließen:

  • AGENTS.md — der Kontext (context): wer wir sind, was wir bauen, und die Hausregeln.
  • Skills — der Prozess (process): die wiederholbaren Schritte, denen das Team folgt, um die Arbeit zu erledigen.

Stellen Sie es sich vor, als würden Sie einen sehr schnellen engineer einstellen. AGENTS.md ist das onboarding Dokument, das Sie ihm am ersten Tag übergeben. Skills sind die runbooks, nach denen sie greifen, wenn sie eine bestimmte Aufgabe erledigen. Dieser Beitrag führt durch beides, anhand eines normalen software development workflows als Beispiel.

AGENTS.md — Die Projektverfassung (Project Constitution)

AGENTS.md ist eine einfache Markdown-Datei im Root-Verzeichnis Ihres repository. AI agents lesen sie automatisch und behandeln sie als permanenten Kontext für jede Aufgabe in diesem Projekt. Es ist kein Ort für Dokumentationen, die bereits anderswo existieren — es ist die kurze Liste von Dingen, die der agent sonst falsch machen würde: Ihr stack, Ihre commands, Ihre conventions und Ihre eisernen Regeln.

Halten Sie es kurz. Eine gute AGENTS.md für ein software project beantwortet vier Fragen:

  • What is this? — Ein Absatz über das product und die architecture.
  • How do I run it? — Die genauen Befehle für build, test, und lint.
  • What are the conventions? — Sprache, Stil, Namensgebung, Ordnerstruktur.
  • What must never happen? — Die nicht verhandelbaren Regeln.

Skills — Der Arbeitsprozess (Work Process)

AGENTS.md teilt dem agent mit, wo er sich befindet. Ein Skill sagt ihm, wie er eine bestimmte Aufgabe ausführen soll. Ein Skill ist eine Markdown-Datei mit einem kleinen frontmatter Header und einem Hauptteil mit step-by-step Anweisungen. Wenn Sie ihn invoken, lädt der agent diesen process und befolgt ihn wie eine checklist — so läuft eine Aufgabe immer gleich ab, egal ob es Montagmorgen oder Freitag 17 Uhr ist, ob Sie am Steuer sitzen oder ein Teamkollege.

Der Trick beim Managen eines Projekts besteht darin, nicht mehr in einmaligen prompts zu denken, sondern in processes. Jede wiederkehrende Aktivität in Ihrem Team — die Entwicklung eines feature, das review eines PR, das Erstellen eines release — ist ein Prozess, dem Sie bereits informell folgen. Schreiben Sie ihn einmal als Skill nieder, und der agent wird ihn für immer konsistent executen.

Skills zu einer Pipeline zusammenfügen

Ein Skill ist nützlich; eine kleine Bibliothek davon macht den agent zu einem Teamkollegen, der Ihren gesamten delivery process kennt. Ein typisches software project endet mit einer Handvoll von skills:

  • feature-dev — ticket → branch → code → tests → PR (siehe oben).
  • code-review — Einen diff lesen und bugs, fehlende tests, sowie Verstöße gegen conventions markieren.
  • bugfix — Den Fehler zuerst durch einen failing test reproduzieren (reproduce), dann beheben, bis er passes.
  • release — Die version erhöhen, das changelog generieren, taggen und release notes entwerfen.

Jeder Skill ist eine Phase in Ihrer pipeline. Da jede Phase dieselbe AGENTS.md liest, teilen sie alle dieselbe Definition von "richtig" — die conventions und Regeln befinden sich an genau einem Ort. Aktualisieren Sie eine Regel in AGENTS.md, und jeder Skill erbt sie beim nächsten Durchlauf.

Wie das in der Praxis funktioniert

Ein Tag Projektarbeit sieht dann so aus:

$ claude

> /feature-dev PROJ-218 "add idempotency key to POST /orders"

  ✓ Restated ticket, listed 4 acceptance criteria
  ✓ Branch feature/PROJ-218-idempotency-key
  ✓ Reused OrderService; added IdempotencyStore
  ✓ Wrote OrdersControllerSpec (4 cases + duplicate-key case)
  ✓ scalafmt clean · 38 tests passed
  ✓ PR description ready

> /code-review

  ⚠ Missing test: concurrent requests with the same key
  ✓ Conventions OK · no secrets · migration untouched

Sie sind auf der Ebene der Absicht (intent) geblieben — "füge einen idempotency key hinzu" — und der agent hat die Mechanik so erledigt, wie es Ihr Team tun würde. Der code-review skill hat sogar eine Lücke gefunden, die der feature skill übersehen hatte, weil die beiden processes unterschiedliche Verantwortlichkeiten kodieren.

Warum dies besser ist als Prompting

  • ✅ Consistency — Der process läuft für jeden, jedes Mal auf dieselbe Weise ab.
  • ✅ Onboarding — Ein neuer Mitarbeiter (Mensch oder agent) ist produktiv, indem er nur zwei Dateien liest.
  • ✅ Single source of truth — Regeln leben in AGENTS.md, nicht in verstreuten Chat-Verläufen.
  • ✅ Living documentation — Ihre AGENTS.md und skills sind die Art und Weise, wie das Team arbeitet, und sie bleiben aktuell, weil sie täglich executen werden.
  • ✅ Scales with the team — Committen Sie sie in das repo, und der gesamte Prozess ist version-controlled.

Fazit (Bottom Line)

Sie führen ein Projekt mit einem AI agent nicht, indem Sie härtere prompts schreiben. Sie managen es, indem Sie zwei Dinge aufschreiben: wer wir sind (AGENTS.md) und wie wir arbeiten (Skills). Sobald diese im repo existieren, beginnt jede Aufgabe damit, dass der agent bereits Ihren stack, Ihre Standards und Ihren process kennt — und Sie können Ihre Aufmerksamkeit auf die Absicht (intent) anstatt auf detaillierte instructions richten.

Fangen Sie klein an: Schreiben Sie noch heute die AGENTS.md und verwandeln Sie dann Ihren am häufigsten wiederholten workflow in einen skill. Fügen Sie den nächsten hinzu, sobald Sie bemerken, dass Sie denselben process zum zweiten Mal erklären.
